Mission

HJB provides access to JMS resources via HTTP

Scope

HJB

  • will allow any language/library capable of communication via HTTP to access JMS-compliant messaging products
  • aims to provide a RESTful equivalent to all of the non-optional portions of the JMS API
  • the access to JMS includes
    • registration of JNDI configured provider resources
    • connection and session management
    • sending/receiving of all types of JMS message

Implementation overview

HJB
  • is deployed as a servlet (HJBServlet) that can run on any compliant Servlet container
  • uses standard Java API libraries and APIs, so as to remain JMS vendor agnostic
  • aims to do one thing very well, i.e., act an HTTP gateway server for JMS resources. It does not concern itself with session management, authentication and security - these features can be added by other HTTP servers in the HTTP request processing chain. This might be the container in which the HJB servlet resides, or a servlet in the same or different container, or any other HTTP proxy server e.g., Apache, Zope, RubyOnRails...
  • is extensively unit tested using JUnit and JMock
  • is fully internationalized
